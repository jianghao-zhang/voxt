# 会议记录

本文档介绍 Voxt 的会议记录模式：它做什么、如何配置、支持哪些 ASR 引擎，以及当前 Beta 的注意事项。

## 功能概览

会议记录是一条独立于普通转录的长时采集流程，适合会议、通话、访谈、播客等场景。

- 使用独立的会议快捷键启动
- 打开的是会议悬浮卡片，不是普通转录卡片
- 不会把文本自动写回当前输入框
- 结果会保存到独立的 `会议` 历史类型
- 可打开详情窗口继续查看、翻译、导出

当前 Beta 使用按音源区分说话方：

- 麦克风标记为 `我`
- 系统音频标记为 `them`

这还不是真正的 diarization。

## 如何开启

1. 打开 `设置 > 通用 > 输出`
2. 开启 `Meeting Notes (Beta)`
3. 授予所需权限：
   - 麦克风
   - 辅助功能 / 输入监控（快捷键链路）
   - 系统音频采集权限

开启后：

- `Hotkey` 页会出现会议快捷键
- `Permissions` 页会出现会议相关权限
- `History` 中会出现会议相关记录与筛选

## 支持的 ASR 引擎

会议记录会跟随当前全局转录引擎。

### Whisper

- 支持
- 使用当前选中的 Whisper 模型
- 跟随现有 Whisper `Realtime` 配置
  - `Realtime ON`：更低延迟的会议更新
  - `Realtime OFF`：质量优先的分段会议转录

### MLX Audio

- 支持
- 使用当前选中的 MLX 模型
- 实时型 MLX 模型会走更低延迟的会议更新
- 其他 MLX 模型走分段会议转录

### Remote ASR

- 支持
- 使用当前选中的 Remote ASR provider 和配置
- 会议里的具体行为按 provider 家族划分：
  - OpenAI：开启 chunk 伪实时预览时仍走现有低延迟分段链路
  - 豆包 ASR：使用单独的 `Meeting ASR` 模型和会议分段 / 文件转写链路
  - 阿里云百炼 ASR：使用单独的 `Meeting ASR` 模型和异步 / 文件转写链路
  - GLM ASR：走上传分段模式

对 `豆包 ASR` 和 `阿里云百炼 ASR`：

- 会议模式不再走 provider 的 live websocket 转写
- 需要在 `设置 > 模型 > Remote ASR > [服务商]` 里单独配置 `Meeting ASR`
- 如果会议模型未配置，Voxt 会阻止启动会议
- 可以用 `Test Meeting ASR` 先验证会议专用请求链路

### Direct Dictation

- 当前 Beta 不支持会议模式

## 会议悬浮卡片

会议卡片偏向长时采集，支持：

- 收起成仅头部模式
- 暂停 / 继续
- 关闭时二次确认
- 带时间戳的段落列表
- 点击段落复制
- 可选的 `them` 实时翻译
- 打开独立详情窗口

如果当前会议已经有转录内容，而你在收起态结束会议，Voxt 会先自动展开，再显示确认对话框，避免误点。

## 实时翻译

会议实时翻译会跟随应用现有的翻译模型 / provider 设置。

- 使用和普通翻译相同的 provider 选择
- 复用相同的 fallback 规则
- 会议模式单独记住自己的目标语言
- 当前 UI 里只翻译 `them` 段落

如果当前翻译 provider 不适合会议里的文本翻译链路，Voxt 会按现有 resolver 规则自动回退。

## 历史记录、详情窗口与导出

正常结束会议时，会依次发生：

1. 关闭会议卡片
2. 保存一条 `会议` 历史
3. 自动打开该会议的详情窗口

详情窗口支持：

- 查看完整会议转录
- 按时间戳定位
- 在已有会议内容上继续翻译
- 在允许的状态下导出

## 注意事项

- 当前 Beta 仍是按音源区分 `我 / them`，不是真正的说话人分离
- 会议模式和普通转录 / 翻译 / 转写是互斥的独立流程
- 会议会写入独立历史类型，并使用独立详情窗口
- 本地模型首次冷启动仍可能需要初始化时间；Voxt 会在悬浮层里显示初始化状态
- 远程 provider 的用户界面保持一致，但底层传输会按 provider 能力切换：
  - `Doubao` / `Aliyun` 的会议模式使用各自的分段 / 文件转写模型
  - 其他 provider 继续走现有的会议分段链路
